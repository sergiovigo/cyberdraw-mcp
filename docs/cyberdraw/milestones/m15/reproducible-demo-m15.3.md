# M15.3 Reproducible Demo - Prompt-to-Diagram MVP

Status: COMPLETE

This runbook records the reproducible M15.3 demo for the Prompt-to-Diagram MVP.
It demonstrates the implemented product boundary:

conceptual user prompt -> client/agent generated Mermaid -> `cyberdraw_create_diagram`
-> internal native `import-mermaid` path -> draw.io page creation -> post-import
page verification.

The server does not receive natural language and does not call an LLM.

## Environment

The reproducible automated demo uses the existing HTTP-local real-environment
harness:

- MCP server created through `createDrawioMcpApp`;
- in-memory MCP client transport for tool invocation;
- local HTTP editor served by the repository;
- real Chromium browser through Playwright;
- real draw.io editor assets;
- real browser plugin loaded into the page;
- real `cyberdraw_create_diagram` MCP tool invocation;
- real draw.io Mermaid import path;
- post-import inspection through public read tools.

This is not an HTTPS/Caddy validation. HTTPS/Caddy remains outside M15.3
closure evidence.

## Demo Scenario

Conceptual prompt supplied to the MCP-capable client or agent:

```text
Create a simple incident response flow: Alert -> Triage -> Investigate -> Contain -> Recover
```

Mermaid generated by the client or agent:

```mermaid
flowchart LR
  A[Alert] --> B[Triage]
  B --> C[Investigate]
  C --> D[Contain]
  D --> E[Recover]
```

Equivalent public tool request:

```json
{
  "format": "mermaid",
  "mermaidType": "flowchart",
  "insertMode": "new-page",
  "mermaid": "flowchart LR\n  A[Alert] --> B[Triage]\n  B --> C[Investigate]\n  C --> D[Contain]\n  D --> E[Recover]",
  "title": "Incident Response Flow",
  "limits": {
    "maxBytes": 12000
  }
}
```

Expected sanitized response shape:

```json
{
  "version": "m15-v1",
  "outcome": "accepted",
  "created": {
    "pageId": "<created page id>",
    "pageName": "Incident Response Flow"
  },
  "safety": {
    "mutatesDiagram": true,
    "mutationAttempted": true,
    "mutationInvocations": 1
  }
}
```

The response must not include XML, raw cells, raw graph data, raw plugin
responses, stack traces, `mermaid_source`, filesystem paths or browser
internals.


## Evidence Checked By The Demo Test

The test in
`packages/drawio-mcp-server/src/real-environment/cyberdraw-create-diagram.test.ts`
verifies:

- a clean test diagram is loaded before the request;
- the public tool name is `cyberdraw_create_diagram`;
- the request uses Mermaid `flowchart`, `insertMode: "new-page"` and
  `limits.maxBytes`;
- the response is `m15-v1` with `outcome: "accepted"`;
- `mutationAttempted` is `true`;
- `mutationInvocations` is `1`;
- exactly one page is created;
- the created page id returned by the response matches the page observed after
  import;
- the created page title is `Incident Response Flow`;
- the created page contains native draw.io vertices and edges produced by the
  Mermaid import;
- no XML or raw plugin/internal payload is present in the public response;
- browser and server logs do not report errors for the operation.

## Cleanup

The harness disposes the MCP client, browser context, browser, HTTP/WebSocket
servers and optional proxy after the test. Each test starts by resetting the
diagram, so the demo is repeatable and does not depend on state left by prior
runs.

## Limitations

- The automated demo is HTTP-local, not HTTPS/Caddy.
- The server expects client-generated Mermaid and does not parse natural
  language.
- Only Mermaid flowcharts are supported by the M15 public wrapper.
- The wrapper creates a new page only; it does not support `target_page`, add,
  replace or embed modes.
- The runtime does not promise rollback or transaction semantics.
- Ambiguous post-dispatch outcomes can return `atomic: "unknown"` and no
  `created` metadata.
- Optional M14 read-only verification can be run separately, but M15 acceptance
  does not depend on M14.
