# M8.1: Real External-Reference Expansion Evidence

## Status

Complete internal milestone evidence.

M8.1 remains private. It does not add a public MCP tool, public schema,
persistence, chunking, streaming, semantic diff, change plans, product
mutations, incremental analysis or stable identity.

## Objective

M8.1 closes the M8 evidence gap by proving in a real draw.io browser runtime
that the hierarchical executor can run a narrow initial scope, observe a
resolvable external reference, derive a second page or layer scope, execute the
second snapshot request, merge both snapshots and build a coherent internal
graph model without using document scope or changing the editor UI.

## Fixture

The real-environment test
`packages/drawio-mcp-server/src/real-environment/hierarchical-snapshot.test.ts`
creates a deterministic synthetic fixture in Chromium-backed draw.io:

- one active page;
- layer `m8-1-focus`;
- layer `m8-1-context`;
- one source rectangle on `m8-1-focus`;
- one target rectangle on `m8-1-context`;
- one edge parented to `m8-1-focus`, with source on the focus layer and target
  on the context layer.

The fixture contains only synthetic labels and generated draw.io IDs.

## Scope Initial

The executor intent is `inspect-layers` with the initial requested scope:

```json
{ "kind": "layers", "pageId": "<active-page>", "layerIds": ["<focus-layer>"] }
```

No document snapshot is requested by the executor.

## External Reference Observed

The first executed layer snapshot includes the focus-layer edge and omits its
target terminal because the target cell is on `m8-1-context`.

The runtime snapshot records `requiresScopeExpansion: true` and an external
reference with:

- source page ID;
- source edge element ID;
- `referenceType: "target"`;
- referenced target element ID;
- referenced page ID;
- referenced layer ID.

The plugin may also record parent references to layer/context ancestors. Those
references are not expansion triggers. The executor only expands `source`,
`target` and legacy `layer` references when they resolve to a concrete page or
layer scope.

## Scope Derived

The executor derives:

```json
{ "kind": "layers", "pageId": "<active-page>", "layerIds": ["<context-layer>"] }
```

The additional step is deterministic:

- ordinal `2`;
- reason `external-context-required`;
- ID prefix `step-02-expansion-`;
- prerequisite equal to the first step ID.

Expansion respects `maxExpansionDepth`, `maxPlanSteps`, hard byte limit,
document identity and freshness checks. It does not derive document scope.

## Second Snapshot

The second snapshot request is sent through the real plugin as a direct
consequence of the first snapshot's external-reference evidence. The test does
not inject the second scope manually and does not use a mocked request handler.

The second scope is a layer scope for the context layer, not document scope. It
returns the referenced target element with a compatible document identity.

## Merge And Graph Result

`mergeScopedSnapshotResults()` combines the focus and context layer snapshots.
It deduplicates elements and layers by contextual page/layer/element key,
preserves page and layer membership, preserves provenance and classifies the
terminal reference as resolved only when the exact target page/layer/element is
materialized by the second snapshot. Ambiguous same-ID targets remain external
references.

The final internal graph model contains:

- source node;
- target node;
- edge relation;
- edge `sourceId` resolved to the source node internal ID;
- edge `targetId` resolved to the target node internal ID;
- no broken-reference findings for this fixture.

Coverage reports the two executed layer targets and `document: false`. The stop
reason is `intent-satisfied`.

## Freshness

For same-scope comparisons, stale detection still compares content revisions.
For cross-scope expansion, scoped `contentRevision` values are not treated as
globally equivalent revisions. The current runtime emits an optional private
`documentRevision` page-anchor signal before scope filtering; when both
snapshots provide it, the executor and merge require it to match. A mismatch
stops execution with `stale-snapshot`, the second result is not merged, and no
automatic replan is attempted.

Older peers may omit `documentRevision`. In that compatibility path, the
executor still validates contract and document identity but cannot prove
cross-scope temporal consistency from document identity alone.

## UI Preservation

The test records active page, selection and editing state before execution and
asserts the same state after execution. The selected source cell remains
selected, the active page is unchanged and editing mode is not left active.

## Limits And Cleanup

The test uses bounded planner and runtime limits, asserts no document scope in
the plan or server snapshot logs, verifies browser and server error logs, and
relies on the real-environment harness cleanup for the MCP client, browser
context, browser, application, HTTP server, WebSocket server and optional proxy.

## Evidence

Focused local evidence:

```sh
pnpm --filter drawio-mcp-server build
NODE_OPTIONS=--experimental-vm-modules ./node_modules/.bin/jest build/real-environment/hierarchical-snapshot.test.js --runInBand -t "external-reference expansion"
```

Observed result:

- 1 real-environment test passed;
- Chromium draw.io runtime used;
- 2 executed layer scopes recorded;
- no document scope requested;
- merged graph relation resolved.

## Risks

- The private external-reference location fields are best-effort runtime
  observations. Older peers may omit them; the executor remains compatible and
  falls back to legacy page/layer-ID resolution.
- Older peers without `documentRevision` remain compatible, but cross-scope
  stale detection is limited to contract and document identity for those peers.
- Cross-page terminal location is still limited by what the active page runtime
  can resolve cheaply. M8.1 proves the required same-page cross-layer case.
- Identity remains provisional and scoped to the merged snapshot.

## Rollback

Rollback removes:

1. optional private `referencedPageId` and `referencedLayerId` fields from
   runtime snapshot external references;
2. executor derivation from those private fields;
3. merge classification of resolved external references;
4. M8.1 real-environment and unit test additions;
5. this document and M8 status updates.

No public API migration or persisted data migration is required.
